## 3. 关键设计决策

### 3.1 命名约定与扫描策略

**改用现有结构,不引入新命名约定**:

CuratedLocalAssetsProvider 扫 `assets/<类别>/<company_id>/` 目录,把以下三类纳入候选:

1. `_raw/<timestamp>_<original>.docx`(主选 — 真实原始文件)
2. `_raw/<timestamp>_<original>.pdf`(次选 — 走 pdf2docx 转换;失败降级 placeholder + .pending_marker)
3. `<resource>.md`(末选 — curated 结构化条目,转 docx 段落作为兜底)

**不**引入 `<asset_type>_<id>_<年份>.docx` 这种命名约定,因为现有摄入流程产出的文件名是 `<timestamp>_<original_name>` 格式,新约定会和摄入流程冲突。

年份过滤通过 manifest 的 year_filter 字段在结果上做(见 §3.4),不依赖文件名 regex。

### 3.2 多命中交互机制(对齐 CLAUDE.md 红线 6)

`lookup()` 方法返回的不是 AssetRef 而是候选列表;b_mode_fill 调用方根据候选数量决定:

```python
def lookup(self, asset_type: str, **kwargs) -> AssetRef:
    candidates = self._scan_candidates(asset_type, **kwargs)
    if len(candidates) == 0:
        return AssetRef(asset_type=asset_type, is_placeholder=True, ...)
    if len(candidates) == 1:
        return candidates[0]
    # ≥2 命中:走交互
    return self._interactive_choose(candidates, asset_type)
```

`_interactive_choose` 行为(对齐 CLAUDE.md 红线 6 "AI 不得代替用户选择"模式):

- stdin 是 tty:打印候选编号 + 路径 + 关键 metadata,让用户输入序号
- stdin 不是 tty(被脚本驱动):报错退出,提示"多命中需用户交互,请改用 --interactive 或在 manifest 显式指定 lookup_priority"
- 提供 `--non-interactive` 模式时,按 manifest.lookup_priority 自动选(见 §3.4)

测试时通过 monkey-patch `sys.stdin.readline` 模拟用户输入。

### 3.3 PDF asset 处理路径

候选方案:

- **Option α(推荐):pdf2docx 在线转换** — 在 resolve() 时若候选是 pdf,调 `pdf2docx.parse()` 转成临时 docx。优点:不污染 _raw 原始归档。缺点:转换有开销 + pdf2docx 偶尔崩溃。
- Option β:摄入流程预转(b_mode_extract 阶段把 pdf 转 docx 入 _raw)— 改动较大,跨脚本协作。
- Option γ:遇到 pdf 直接返回 placeholder + .pending_marker — 简单但损失 pdf asset 的可用性。

**我的选择:Option α**。失败时 fallback 到 Option γ,加日志告警。

requirements.txt 增加 pdf2docx 注释("仅 V3-1 PDF asset 自动转换需要")。

### 3.4 manifest.yaml 字段扩展

`b_mode_extract.py` 产出的 manifest 增加(可选)字段:

```yaml
assembly_order:
- section_id: (一)
  section_title: 营业执照
  asset_type: 资质证书
  source_type: asset_lookup
  source: 申请人资格要求 3:营业执照
  asset_query:
    type: 资质
    name: 营业执照
  # === V3-1 新增字段(可选,缺省按默认行为)===
  lookup_priority: latest_year_first   # latest_year_first(默认) / name_match_first / review_status_first
  year_filter: ">=2024"                # 可选,语义:>=YYYY / <=YYYY / YYYY 单年
```

**lookup_priority 语义**:

- `latest_year_first`(默认):多命中时按文件 mtime 或 _raw 文件名 timestamp 排序,最新优先
- `name_match_first`:候选 metadata.name 与 spec.asset_query.name 完全匹配的优先
- `review_status_first`:curated .md 的 review_status=approved 的优先(摄入流程标记)

**year_filter 语义**:

字符串形式表达 year 过滤,语法子集:`>=YYYY` / `<=YYYY` / `YYYY` / `YYYY-YYYY`(范围)。年份从 _raw 文件名 timestamp 或 curated .md 的 `颁证日期` / `合同签订日期` 字段抽。

向后兼容:无 lookup_priority / year_filter 字段时按 latest_year_first 默认 + 不过滤。

**注意**:本 Plan 只让 CuratedLocalAssetsProvider **读** 这两个字段;**不**修改 b_mode_extract.py 让它产出这两个字段(b_mode_extract 改动属计划外,需另启动)。manifest 现有产出无这两字段时走默认。
