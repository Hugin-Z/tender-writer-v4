## 2. 实现设计

### 2.1 分流判断在哪一步

**当前 `c_mode_run.run_part`**(L94-141)流程:
1. 检查 production_mode == 'C'
2. 可选 extract-text
3. 检查 intermediate.json 存在
4. **build-from-json**(产 template.docx + variables.yaml)— 走 jinja
5. **c_mode_fill**(填变量)— 走 jinja

V3-7 在第 3 步检查后、第 4 步之前,插入"尝试 docx passthrough"。

### 2.2 判断规则的精确边界

**精确边界**(全部在 `c_mode_docx_passthrough.passthrough_part` 内已实现,我**不重写** ):

1. `brief.source_meta.source_format == 'docx'`
2. `part.production_mode == 'C'` 且 `part.sub_mode == 'C-template'`
3. `part.source_anchor.type == 'text'`
4. `len(variables.yaml.variables) <= max_vars`(默认 5)

**不引入扩展名 / 自定义字段判断**:passthrough_part 已经做了完整判定,c_mode_run 只需"调用 + 看返回"即可。

### 2.3 调用方式 + 优先级

**最简实现**:c_mode_run 直接 `import passthrough_part` 调用(同进程),不走 subprocess:

```python
# 在 run_part 中,intermediate.json 检查通过后:
from c_mode_docx_passthrough import passthrough_part
ps_result = passthrough_part(project, part_idx, max_vars=5, dry_run=False)
if ps_result['status'] == 'ok':
    print(f"[Part {part_idx}] {name} - docx 直切完成 ({ps_result['msg']})")
    return {'status': 'ok', 'msg': f'docx 直切: {ps_result["msg"]}'}
elif ps_result['status'] == 'error':
    return {'status': 'error', 'msg': f'docx 直切错误: {ps_result["msg"]}'}
# status == 'skipped' → fallthrough 到原 jinja 流程
print(f"[Part {part_idx}] {name} - docx 直切跳过 ({ps_result['msg']});走 jinja 模板填充")

# 原 jinja 流程(build-from-json + fill)继续...
```

**优先级:passthrough 优先,不适用时 fallthrough 到 jinja**。理由:
- passthrough 适用条件严格(4 个 AND),不会误用
- passthrough 跳过 jinja 模板替换,产物更接近招标原文
- 不适用时 passthrough_part 返回 `skipped`(带 reason),c_mode_run 顺势走 jinja

**不需要新加 CLI flag**:c_mode_run 现有签名 `--project --part --all --with-extract --non-interactive --entity-id` 全部保留;passthrough max_vars 写死 5(passthrough_part 默认值),不暴露 CLI 控制。如果未来需要可调,加 `--passthrough-max-vars` flag(本期不做,留 V4)。

### 2.4 日志输出区分两种路径

新增的两条日志行:

- 直切成功:`[Part {idx}] {name} - docx 直切完成 ({passthrough msg})`
- 直切跳过 / fallthrough:`[Part {idx}] {name} - docx 直切跳过 ({reason});走 jinja 模板填充`

汇总段(已有的 print 段)不动,仍用 OK / SKIP / ERR icons。每个 Part 的 `msg` 字段补上"docx 直切" / "jinja" 字样。

**grep 自检**:`grep "docx 直切" demo_run.log` 应 ≥1;`grep "jinja 模板填充" demo_run.log` 应 ≥1。

### 2.5 .reviewed 闸门 / bidding_entity 等其他逻辑

不动。c_mode_run 的 `_ensure_bidding_entity` + `load_parts` + 主入口 / `--all` / `--non-interactive` 全部保留。V3-7 只在 `run_part` 内插入分流。

## 3. 测试设计(不补单元 test)

**不补单元 test**,理由:

- 分流逻辑 ≤ 10 行,核心是 `if status == 'ok' / 'error' / 'skipped'` 三态判断,trivial
- `passthrough_part` 内部 4 条判断在原代码已就位,V3-7 不修改其行为
- 单元测试需要 mock subprocess + 构造 brief.json + 构造 source docx,fixture 准备 cost ≫ 测试价值
- demo 端到端验证(C1+C2+C3)更有信息量:demo 项目 6 个 C 模式 Part,4 个走 passthrough(vars≤5)+ 2 个走 jinja(vars>5),自然覆盖两条路径

## 6. 风险

| 风险 | 缓解 |
|---|---|
| `passthrough_part` 在 dry_run=False 时若 `_raw_lines_to_docx_para_index` 失败(返回 -1),返回 status=error,c_mode_run 退出 | 这是 passthrough_part 自身的错误处理;c_mode_run 看到 error 时 fail 不 fallthrough 是合理的——passthrough 已选定但执行失败,fallthrough 到 jinja 会掩盖 bug。仅在 status=skipped 时才 fallthrough |
| variables.yaml 不存在时 passthrough_part 默认 n_vars=0 → 通过阈值检查 → 走 passthrough | 这是 passthrough_part 已有行为,本期不改。但 c_mode_run 流程中 build-from-json **还没跑**,所以 variables.yaml 必然不存在 → 默认 n_vars=0 → 所有 C-template+anchor.text 都会走 passthrough。**这会导致 demo 验证失败**:Part 1/2 即使 vars=9/7 也会被错误归为 passthrough 路径! |
|   ↑ **修复方案**:passthrough_part 调用前,c_mode_run 主动从 intermediate.json 读 variables 数量;若 intermediate 含 vars>5 → 直接判 jinja,不进 passthrough 调用 | 见 §2.3 实际实现需调整:从 intermediate.json[blocks 各 block 的 variables 数量合计]读 |

**风险 2 是 §2.3 设计的一个真实漏洞**,我已识别。修正方案:

```python
# 改进的 §2.3 实现:
# 1. 从 intermediate.json 读 variables 数量(因为 build-from-json 还没跑,variables.yaml 不存在)
inter_data = json.loads(inter.read_text(encoding='utf-8'))
n_vars = sum(len(b.get('variables', [])) for b in inter_data.get('blocks', []))
# (或 inter_data.get('variables', [])  ← 取决于 intermediate.json schema,需进一步确认)
PASSTHROUGH_MAX_VARS = 5

if n_vars > PASSTHROUGH_MAX_VARS:
    # 已知不适用,直接走 jinja
    print(f"[Part {part_idx}] {name} - 变量数 {n_vars} > {PASSTHROUGH_MAX_VARS},走 jinja")
else:
    # 尝试 passthrough(它内部还会检查其他 3 条)
    from c_mode_docx_passthrough import passthrough_part
    ps_result = passthrough_part(project, part_idx, max_vars=PASSTHROUGH_MAX_VARS, dry_run=False)
    ...
```

**Phase 2 期间需要先确认 intermediate.json 的 schema**,看 variables 是顶层字段还是 blocks 内嵌。这是 plan 落地前的小核查,应在写代码前先 .venv python 看一眼。
