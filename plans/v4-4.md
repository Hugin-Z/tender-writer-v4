# plans/v4-4.md — V4-4 C-attachment 真实现

## 任务定义

C-attachment(扫描件附件,整份文件单独挂,不进主响应文件文本流)当前实现到 schema 层 + skip 行为测试,业务层 0 真实现(5 处 raise/skip)。本任务把 5 处执行端的 raise/skip 改成真实现 + 改造 3 处 stale 测试,使 C-attachment 端到端跑通。

## 4 个设计裁定(已定 / Phase 2 据此实现)

- 裁定 C — 边界:B 模式素材内容融进文本流(assembled.docx);C-attachment 整份文件不进任何文本流,作为独立附件单独挂。同一份文件按用途分流,两者可并存不重叠。
- 裁定 B — 附件源:从本仓库现有 assets 路径取,复用 CuratedLocalAssetsProvider 抽象(与 B 模式同一 assets 仓库,取用方式不同:B 取内容、C-attachment 取整文件)。不接外部知识库 / 外部源(那是 D 模式范畴,V4-4 不碰)。
- 裁定 A — 占位策略:附件源文件不存在时,产占位 + 显式标记,不硬失败(部分附件本就需用户人工放置)。R10 红线:占位必须显式可见 —— 产物清单标 __PENDING_USER__(或实测确认的现有占位字面)+ operations_checklist.md 列出"待人工放置附件"+ export 透传占位状态。绝不静默产空附件文件假装齐备。
- 裁定 D — 布局:attachment 产物进 final_tender_package/attachments/<part_name>/ 子目录(与 C-reference 不进主文件、ops_checklist 留说明 同构)。"读招标文件装订要求结构化 + 消费"经实测确认当前机制不存在(parse_tender 标位置未抽内容、无字段、无下游消费),不在 V4-4 范围,另立 V4 候选项登记。

## 实现指令(Phase 2 据实参照,不凭印象编码)

通则:5 处业务改动每处都有 C-template 或 C-reference 同型分支可参照。Phase 2 落每处前,先读该文件里 C-template + C-reference 的对应分支实现,据其同型结构落 C-attachment 分支。不脱离现有代码结构自创写法。每处落盘前实测确认当前确切字面 + 行号(行号以实测为准,用字面 anchor)。

### S1 — migrate_brief_schema.py:迁移透传

参照 C-template / C-reference 在本文件的透传写法,把 C-attachment 从 raise NotImplementedError 改为同型透传(brief 旧 schema 含 sub_mode='C-attachment' → 新 schema 原样保留)。

### S2 — c_mode_extract.py:attachment extract 分支

参照 C-reference 的 _build_instructions_md 分支结构。C-attachment 分支产出 intermediate.json,内容含附件清单:每个附件的 company_id / 类别 / 源文件路径(从 assets 解析)/ 目标文件名。源文件不存在的条目标占位(裁定 A:__PENDING_USER__ 或实测确认的现有占位字面)。此阶段只产清单,不拷贝文件(拷贝在 fill 阶段)。

### S3 — c_mode_fill.py:校验/占位/拷贝分支

参照 C-reference 的 instructions.md 存在性校验分支 + C-template 的 filled.docx 产出分支。C-attachment 分支:读 extract 的 intermediate.json 附件清单 → 逐条校验源文件存在性 → 存在则拷贝到 part 的 attachments 工作目录 → 不存在则保留占位标记(不拷贝、不报错、不产空文件)。产物 = 实际附件文件副本(存在的)+ 占位记录(缺失的)。

### S4 — v45_merge.py:入 merge_order + 拷贝

参照 C-reference 的 kind='c_reference' 入 merge_order + ops_checklist 段落写法。C-attachment:kind='c_attachment' 入 merge_order;主循环不 composer.append(不进 final_response.docx);把 fill 阶段的附件副本拷贝到 final_tender_package/attachments/<part_name>/;在 operations_checklist.md 加附件说明段(列本 Part 的附件清单 + 占位项明确标"待人工放置")。

### S5 — export_deliverables.py:映射分支

参照 C-reference 的映射分支。增 C-attachment 分支:映射 attachment 目录到"C 模式产出/NN_<name>(附件)/"。占位状态透传(缺失附件在 export 产物里仍标占位,不假装存在)。mode 若现有 mapping 结构支持目录级拷贝则用之,否则参照现有最接近的 mode 字面,实测确认。

### S6 — 测试改造 3 处

- test_v45_merge.py case_5:从"验整项跳过"改为"验 C-attachment 入 merge_order + 拷贝到 attachments/ + ops_checklist 有附件段"。断言用真字段(merge_order 含 c_attachment kind / attachments 目录产出 / ops_checklist 含附件清单),禁泛断言。
- test_export_deliverables.py ok_skip_cattach:从"验不映射"改为"验映射 attachment 目录 + 占位透传"。
- fixture brief_c_attachment_skip.json:改名去 _skip 后缀(如 brief_c_attachment.json),内容对齐"真实现"语义;含 clean(附件源存在)+ 至少一个占位(附件源缺失)两种条目,验占位路径。改名后全仓 grep 旧文件名 0 残留引用。

### S7 — changelog + 候选项登记

docs/changelog.md 加 V4-4 entry(字面 Phase 2 落此 commit 前停下,等 chat 端起草)。同 entry 登记独立候选项:"招标文件投标文件构成与装订要求结构化抽取 + 下游消费(当前 parse_tender 仅标位置未抽内容、无字段、无消费),不在 V4-4,留 V4 后期"。
