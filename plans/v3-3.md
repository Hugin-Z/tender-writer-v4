## 0. 数据用途(必须先想清楚)

> 这一节直接回应"埋点必须交代用途"的硬规则,没想清楚用途的埋点不做。

### 0.1 谁是消费者

主要消费者:**用户(开发者本人)** 在两类场景查 timing 数据:

1. **plan 估算校准**:V3-N+ 写 plan 时引用历史 timing("parse_tender 实测 X 秒,本次 V3-9 加 4 项检查后预估 Y 秒")
2. **慢阶段定位**:跑完整 demo 后,grep timing_report 找最贵的阶段,优先级排队优化

次要消费者:Phase 3 review 报告引用 timing 数据("本 V3-N 跑全 demo 共 X 秒,与 V3-(N-1) 对比 Y 秒")。

### 0.2 触发什么决策

具体使用场景(每条都给出实际触发动作):

| 场景 | 触发条件 | 决策动作 |
|---|---|---|
| **慢阶段优化排队** | timing_report 显示某阶段 duration > 30s | 记入 docs/v3_planning.md 或 v4_planning.md "优化候选"清单,排优先级 |
| **回归检测** | 某 commit 后某阶段 duration 比 baseline +50% | Phase 3 review 时报告告警,确认是否预期 |
| **plan 估算校准** | 写 plan 时引用历史 duration | 替换"X 分钟(墙钟估算)"为"X 秒(实测,_timing.json L#N)" |
| **demo 端到端节奏** | demo 全跑 > 3 分钟 | 触发"是否拆 demo / 减少阶段串行"讨论 |
| **PDF 解析卡顿提示** | parse_tender > 30s | (本期不做自动告警,但 timing_report 标红显眼,让用户主动看 FAQ Q4 OCR 流程) |

### 0.3 数据格式与查询路径

- **存储**:每个项目独立文件 `projects/<project>/output/_timing.json`,以 append 模式记录每次脚本调用
- **查询**:`./run_script.bat timing_report.py --project <name>` → 控制台打印 markdown 表 + 写 `output/_timing_report.md`
- **粒度**:阶段级(整个脚本一次调用 = 一条 entry),不细到函数级(那是 V4+ profiler 范畴)
- **保留策略**:append 不截断(历史保留,timing_report 默认显示最近 10 次 + 全期统计)

### 0.4 不在 V3-3 范围的(显式排除)

- 自动告警 / 阈值触发动作:本期只采集 + 报表,不主动提示
- 函数级 profiler:用 cProfile / scalene 等专业工具,本期不重做轮子
- 跨项目对比:本期 _timing.json 单项目独立,不聚合
- CI 集成:本期不接 GitHub Actions,数据停在本地

如果上面 4 条某天有需求,V4 / V5 做。

## 2. 实现设计

### 2.1 scripts/_timing_hook.py(基础设施)

```python
"""
_timing_hook.py · 阶段耗时埋点上下文管理器(V3-3)

用法:
    from _timing_hook import stage_timer
    
    def main():
        with stage_timer("parse_tender", project_dir):
            # ... 原 main 逻辑 ...
            pass
"""
import json
import time
from contextlib import contextmanager
from datetime import datetime
from pathlib import Path


@contextmanager
def stage_timer(stage_name: str, project_dir: Path | None = None):
    """记录一次脚本调用的耗时,append 到 project_dir/output/_timing.json。
    
    project_dir 为 None 时跳过写入(脚本无 --project 上下文 fallback)。
    异常路径:即使 with 块抛异常,也会记录 success=False 然后 re-raise。
    """
    started_at = datetime.now().isoformat(timespec='seconds')
    t0 = time.perf_counter()
    success = True
    error_msg = ""
    try:
        yield
    except Exception as exc:
        success = False
        error_msg = repr(exc)[:200]
        raise
    finally:
        duration = time.perf_counter() - t0
        ended_at = datetime.now().isoformat(timespec='seconds')
        if project_dir is not None:
            _append_entry(project_dir, {
                "stage": stage_name,
                "started_at": started_at,
                "ended_at": ended_at,
                "duration_seconds": round(duration, 3),
                "success": success,
                "error": error_msg,
            })


def _append_entry(project_dir: Path, entry: dict):
    out_dir = project_dir / "output"
    out_dir.mkdir(parents=True, exist_ok=True)
    timing_path = out_dir / "_timing.json"
    if timing_path.exists():
        try:
            data = json.loads(timing_path.read_text(encoding='utf-8'))
        except json.JSONDecodeError:
            data = {"entries": []}
    else:
        data = {"entries": []}
    data["entries"].append(entry)
    timing_path.write_text(
        json.dumps(data, ensure_ascii=False, indent=2),
        encoding='utf-8',
    )
```

**设计点**:

- **append 模式**:历史保留,不截断。同一脚本多次跑都记录,timing_report 聚合时显示 last/avg/min/max。
- **project_dir=None 时静默跳过**:某些场景脚本没有项目上下文(如未来一次性工具),不强制写。
- **异常也记录**:即使脚本 fail,也写一条 success=False entry,便于诊断"哪次失败、当时跑了多久"。
- **timing 文件独立**:`_timing.json` 名字带下划线前缀,不属于业务产物;不进 tracked_outputs;可被 .gitignore 忽略(见 §2.5)。
- **timestamp 用 ISO format**:可读 + 易解析。

### 2.2 9 个脚本主入口包装

每个脚本 `main()` 函数体的开头加:

```python
from _timing_hook import stage_timer
# ... 解析 args ...
project_dir = _infer_project_dir(args)  # 见下
with stage_timer("<stage_name>", project_dir):
    # 原 main 逻辑全部缩进进 with 块
    ...
```

**stage_name 命名约定**:用脚本文件名(去 .py),如 `parse_tender` / `b_mode_fill`。简单直观,grep 友好。

**project_dir 推断逻辑**(每个脚本独立):

| 脚本 | project_dir 推断 |
|---|---|
| parse_tender | `Path(args.out).parent`(args.out 是 output/,parent 是 project/) |
| build_scoring_matrix | `Path(args.brief_path).parent.parent`(brief.json 在 project/output/) |
| generate_outline | `Path(args.matrix_csv).parent.parent` |
| c_mode_run | `ROOT / 'projects' / args.project` |
| b_mode_extract / b_mode_fill | `ROOT / 'projects' / args.project` |
| v45_merge | `ROOT / 'projects' / args.project` |
| check_cross_consistency | `Path(args.brief).parent.parent` |
| compliance_check | `Path(args.matrix_csv).parent.parent` |
| export_deliverables | `ROOT / 'projects' / args.project` |

抽 helper `_infer_project_dir(args)` 到 `_timing_hook.py`?**不抽**,因为每个脚本的 args 字段不同,strategy pattern 反而绕。每个脚本自己 1-2 行推断。

### 2.3 scripts/timing_report.py(报表生成)

```python
"""
timing_report.py · 把 _timing.json 聚合为 markdown 报表(V3-3)

用法:
    ./run_script.bat timing_report.py --project demo_cadre_training
"""
import argparse, json, statistics
from pathlib import Path

ROOT = Path(__file__).resolve().parent.parent

def render_report(entries: list[dict]) -> str:
    """聚合 entries 按 stage,产出 markdown 表 + 最近 10 次详单。"""
    # 按 stage 分组
    by_stage = {}
    for e in entries:
        by_stage.setdefault(e['stage'], []).append(e)
    
    lines = ["# Timing Report\n"]
    lines.append("## 阶段聚合统计\n")
    lines.append("| 阶段 | 调用次数 | 最近 (s) | 平均 (s) | 最小 (s) | 最大 (s) | 失败次数 |")
    lines.append("|---|---|---|---|---|---|---|")
    for stage in sorted(by_stage.keys()):
        es = by_stage[stage]
        durations = [e['duration_seconds'] for e in es]
        n_fail = sum(1 for e in es if not e.get('success', True))
        last = durations[-1] if durations else 0
        avg = statistics.mean(durations) if durations else 0
        lines.append(
            f"| {stage} | {len(es)} | {last:.2f} | {avg:.2f} | "
            f"{min(durations):.2f} | {max(durations):.2f} | {n_fail} |"
        )
    
    lines.append("\n## 最近 10 次调用\n")
    lines.append("| 时间 | 阶段 | 耗时 (s) | 状态 |")
    lines.append("|---|---|---|---|")
    for e in entries[-10:]:
        status = "✓" if e.get('success', True) else f"✗ ({e.get('error','')[:40]})"
        lines.append(f"| {e['started_at']} | {e['stage']} | {e['duration_seconds']:.2f} | {status} |")
    
    return "\n".join(lines) + "\n"

def main():
    p = argparse.ArgumentParser()
    p.add_argument('--project', required=True)
    args = p.parse_args()
    timing_path = ROOT / 'projects' / args.project / 'output' / '_timing.json'
    if not timing_path.exists():
        print(f"[错误] 未找到 {timing_path};是否跑过任何阶段脚本?", file=sys.stderr)
        sys.exit(1)
    data = json.loads(timing_path.read_text(encoding='utf-8'))
    report = render_report(data.get('entries', []))
    out_path = timing_path.parent / '_timing_report.md'
    out_path.write_text(report, encoding='utf-8')
    print(report)
    print(f"\n[完成] 报表已写入 {out_path}")
```

### 2.5 .gitignore 处理

`_timing.json` / `_timing_report.md` 是 demo 跑产生的中间数据,会变化。是否 git track?

- **track**:V3-3 完工时把 demo 的 _timing.json 入库,作为 baseline reference,后续 V3-N 完工时贴对比
- **不 track**:每次跑 demo 都会改,git diff 噪音

**我倾向 track**(理由:V3-3 的 plan §0.2 用途列了"回归检测"和"plan 估算校准",需要可复现 baseline。每次 V3-N 完工时手动覆盖 demo 的 _timing.json 入库,作为下次的对比基准)。

写 .gitignore 时不加 `_timing.json` exclusion(默认 demo 项目下的会被 track,因为 `!projects/demo_cadre_training/` exception)。
