## 0. Phase 0 关键发现(必须先决策)

拆 3 个现有 fixture docx + demo final_response.docx 的 `word/fontTable.xml`,**全部含相同 8 个 font 声明**:

```text
['Symbol', 'Times New Roman', 'Cambria',
 'ＭＳ 明朝',           ← 日文字体!
 'Calibri',
 'ＭＳ ゴシック',        ← 日文字体!
 'Courier', 'Arial']
```

**根因**:`python-docx` 库的默认 boilerplate 自带 fontTable,无论用户实际用没用过这些字体。**所有用 python-docx 生成的 docx 都有这 8 项**——包括我们的 b_mode_fill / c_mode_fill / docx_builder 产物。

### 设计影响

如果 V3-8 简单实现"fontTable 含非白名单字体即报",**所有产物都会误报**(systematic false positive)。

### 三条候选路径

**Path A(推荐)—— 容忍清单 + 增量报警**

- 维护两个清单:
  - **白名单**(已有 V3-2 检查的扩充):宋体/黑体/仿宋/微软雅黑 中文族 + Times New Roman/Arial/Calibri/Cambria 英文族
  - **boilerplate 容忍清单**(V3-8 新):MS 明朝 / MS ゴシック / Symbol / Courier(python-docx 默认)
- fontTable 级检查报警条件:`字体 ∉ 白名单 ∧ 字体 ∉ boilerplate 容忍清单`
- 中文别名归一化:`SimSun` / `宋体-简` / `NSimSun` 都视作`宋体`,通过 alias map

**Path B —— fontTable 级仅报"被引用且非白名单"**

- 只报 run/style 实际引用的非白名单字体(其实就是 V3-2 已做的段落级检查的扩展)
- 缺点:V3-8 没新价值,fontTable 解析本身退化为 noop

**Path C —— 同 Path A,但同时在 docx_builder 主动清理 fontTable boilerplate**

- 既加 V3-8 检查,也修源头(让 docx_builder 把 MS 明朝 / MS ゴシック 从 fontTable 删除)
- 缺点:范围扩大,docx_builder 改动属另一项工作

**我的推荐**:**Path A**。

理由:
- 容忍清单透明记录 python-docx 默认 boilerplate,V3-8 只关心"额外加入的可疑字体"——这才是 V3-8 的增量价值
- 不动 docx_builder(避免范围漂移);如果未来要清理 boilerplate,可登记 V4 或 V3-13(类似 V3-3 衍生 V3-13 模式)
- 中文别名归一化是必需的(SimSun = 宋体)

def check_font_safety(docx_path: Path) -> list[str]:
    """检查 docx 字体安全。

    2. (V3-8 新增) word/fontTable.xml:warn(声明非白名单且非 boilerplate
       的字体,潜在 fallback 风险)

    返回 issues 列表(混合 fail/warn,前缀区分),空表示通过。
    """
    issues = []

    # ── 现有段落级检查不动 ──
    # ... existing V3-2 logic for Normal style + run-level rFonts ...

    # ── V3-8 新增:fontTable.xml 级检查 ──
    fonttable_issues = _check_fonttable(docx_path)
    issues.extend(fonttable_issues)

### 2.2 fontTable.xml 解析

使用 `xml.etree.ElementTree`(stdlib,无新依赖):

```python
def _check_fonttable(docx_path: Path) -> list[str]:
    import zipfile
    import xml.etree.ElementTree as ET

    NS = '{http://schemas.openxmlformats.org/wordprocessingml/2006/main}'

    try:
        with zipfile.ZipFile(docx_path) as z:
            if 'word/fontTable.xml' not in z.namelist():
                return []  # 没 fontTable,不检查
            xml_text = z.read('word/fontTable.xml').decode('utf-8')
    except (zipfile.BadZipFile, KeyError):
        return ['字体检查(fontTable):docx 解压失败']

    try:
        root = ET.fromstring(xml_text)
    except ET.ParseError:
        return ['字体检查(fontTable):xml 解析失败']

    fonts = [f.get(f'{NS}name') for f in root.findall(f'{NS}font')]
    fonts = [f for f in fonts if f]

    issues = []
    suspicious = []
    for font in fonts:
        normalized = _normalize_font_alias(font)
        if normalized in ALLOWED_ALIASES_NORMALIZED:
            continue
        if normalized in BOILERPLATE_TOLERATED:
            continue
        suspicious.append(font)

    if suspicious:
        issues.append(
            f'字体检查(fontTable warn):docx 声明非白名单字体 {suspicious},'
            f'非 python-docx 默认 boilerplate;若被实际引用可能导致 WPS '
            f'fallback 渲染问题。白名单 + boilerplate 见 compliance_check.py 顶部注释。'
        )
    return issues
```

### 2.3 白名单 + 别名映射 + boilerplate 容忍清单

**白名单(中文族 + 英文族,含别名)**:

```python
# 中文字体规范名(归一化目标)
CHINESE_CANONICAL = {
    '宋体':     ['宋体', 'SimSun', '宋体-简', 'NSimSun', 'STSong', '宋体-繁'],
    '黑体':     ['黑体', 'SimHei', 'STHeiti'],
    '仿宋':     ['仿宋', 'FangSong', '仿宋_GB2312', 'STFangsong'],
    '微软雅黑': ['微软雅黑', 'Microsoft YaHei', 'Microsoft YaHei UI'],
}

# 英文字体规范名(直接列名,无强别名映射)
ASCII_CANONICAL = ['Times New Roman', 'Arial', 'Calibri', 'Cambria']

# 归一化后的所有合法字体(查找用 set)
ALLOWED_ALIASES_NORMALIZED = set(CHINESE_CANONICAL.keys()) | set(ASCII_CANONICAL)

# python-docx 默认 boilerplate 自带的字体(知道存在但不报)
BOILERPLATE_TOLERATED = {
    'Symbol',         # 默认符号字体
    'Courier',        # 默认等宽
    'ＭＳ 明朝',       # 默认日文衬线(MS Mincho 全角)
    'ＭＳ ゴシック',   # 默认日文黑体(MS Gothic 全角)
}


def _normalize_font_alias(font_name: str) -> str:
    """把字体别名归一化到 canonical name。"""
    for canonical, aliases in CHINESE_CANONICAL.items():
        if font_name in aliases:
            return canonical
    if font_name in ASCII_CANONICAL:
        return font_name
    return font_name  # 未识别,原样返回(由调用方判定)
```

### 2.4 段落级 vs fontTable 级关系

**两层并存,不替代**:

| 层 | 报警等级 | 触发条件 | 渲染影响 |
|---|---|---|---|
| 段落级(V3-2 已有) | **fail** | Normal 样式或 run 级 rFonts 引用非白名单 eastAsia | 直接渲染异常(WPS fallback 到日文) |
| fontTable 级(V3-8 新) | **warn** | fontTable.xml 声明非白名单且非 boilerplate 字体 | 潜在 fallback 风险(取决于是否被引用) |

**为什么 warn 不 fail**:fontTable 声明字体不一定被实际引用;若 Normal 样式正确,即便 fontTable 有可疑字体,也不渲染问题。报 warn 让用户知情,不阻塞流程。

报告输出在同一 `check_font_safety` 返回的 issues list 里,前缀 `字体检查(段落 fail)` vs `字体检查(fontTable warn)` 区分。

### 2.5 误报控制(对齐 V3-4 经验)

V3-8 同样 fix 三档 escape:

1. **CLI 调白名单**:`--font-allow-extra "MyCustomFont,SomeOther"` 加临时白名单(Phase 2 决定 是否加 flag,默认不加,V4 再加)
2. **boilerplate 清单是 hardcode**(因为 python-docx 默认行为稳定,不需用户调)
3. **行级 nolint**:**不引入**(check_font_safety 是 docx 二进制级检查,无法在 docx 内嵌注释跳过;只能整 docx 级 escape)

实际上,V3-8 设计已经把"被 python-docx 默认带入的字体"全部容忍掉了,真正触发误报的只有"用户手工加 / 第三方插入的非白名单字体"——这种情况确实需要用户处理(改字体或 cleanup docx),不是检测过严。

**误报率监控**:

- demo final_response.docx + tender_response.docx + assembled.docx 全部跑一遍,断言 fontTable level 0 warn(都是 python-docx 产出,boilerplate 容忍清单覆盖)
- 如果 demo 出现 fontTable warn,说明 boilerplate 容忍清单不全,Phase 2 commit 内补全
- 如果补全后仍 warn,说明 demo 数据有非默认字体——真实风险,不是误报,保留 warn
