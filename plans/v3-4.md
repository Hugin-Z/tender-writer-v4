## 2. 实现设计

### 2.1 函数签名

新增 `check_content_keyword_stitching(text, matrix_rows, args)` 到 `scripts/check_chapter.py`,作为检查项 [8]:

```python
def check_content_keyword_stitching(
    text: str,
    matrix_rows: list[dict],
    window: int = 30,
    threshold_warn: int = 4,
    threshold_fail: int = 5,
) -> tuple[str, str]:
    """V3-4: 正文级缝合句检测(扩展 check_title_keywords 的标题级 cheat 检测)。

    返回 (status, detail)。status: pass/warn/fail。

    算法:
    1. 收集所有 scoring_matrix 第 5 列关键词
    2. 对正文段落(跳过表格/列表/标题/代码块/nolint 注释行)做滑窗
    3. window 字内命中 ≥threshold_warn 关键词 → warn
       命中 ≥threshold_fail 关键词 → fail
    4. 标记到 detail 含行号 + 命中关键词
    """
```

集成到 `main()`: 列表序号 [8],与既有 7 项并列。`--legacy-mode` 下 fail 降级为 warn(对齐既有 v2 升级降级机制)。

### 2.2 阈值与误报控制(核心节,直接回应用户三问)

**问 1:为什么是 4(warn) / 5(fail) 不是 3 / 6?**

依据:

- **既有标题级 cheat 检测**(check_title_keywords L142-156)用阈值 **3**(标题 ≤30 字内 ≥3 关键词 跨度 ≤20)。标题密度天然高于正文,3 在标题合适,搬到正文会过严(用户提示的"质量管理体系覆盖审核工作程序"4 关键词在 14 字,正常专业表述)。
- 实证测试:扫 `projects/demo_cadre_training/output/chapters/` 6 章 markdown,把"4 关键词 30 字"试跑,看哪些段落被命中。如果合规段落被命中比例 > 10%,提阈值。
- 行业 lint 工具参照:markdownlint 类规则的"3-strikes" 模式(2 次 warn,3 次 fail);本期 V3-4 套用 4 warn / 5 fail 双档,留 1 个关键词的容差。
- **本期定 4 warn / 5 fail 是初值**,如果 demo 实测发现误报多,Phase 2 期间向 5/6 调,Phase 3 review 报告里说明最终阈值。

**护栏(用户审 plan 时加)** — 阈值调整不是无限的实施细节:

- demo 6 章烟测**≥3 段被错误标 fail**(误报率 ≥50%)时触发调整决策
- **第一次调整(4/5 → 5/6)** :Phase 2 内自行调整,commit message 注明,继续走完
- **仍不达标(5/6 → 6/7 或更宽)**:**停下汇报**,这是设计决策不是实施细节;算法本身可能不足以判别缝合,需讨论换思路(加连接词检测 / 句法分析 / 人工 review),不允许无限放宽阈值
- **理由**:无限放宽阈值是检测规则的死亡之路 — 阈值放到 100 就永远不命中,看似"误报零"实际"检测无效",真问题被参数掩盖

**问 2:反例 fixture 怎么准备?**

测试 fixture 至少含:

| 类型 | fixture 内容 | 期望 |
|---|---|---|
| 正例 1(典型缝合) | "本方案审核质量控制措施风险管理制度工作程序标准方法保障落实"(7 关键词 25 字,无连接词) | fail |
| 正例 2(中度密集) | "项目管理体系包含质量审核技术规范四个模块"(4 关键词 18 字,有"包含"但仍可疑) | warn |
| 反例 1(用户原型) | "本方案的质量管理体系覆盖审核与工作程序"(4 关键词 19 字,合规专业表述) | pass |
| 反例 2(列表枚举,跳过规则) | "- 质量管理\n- 审核流程\n- 工作程序\n- 风险控制"(列表 4 条) | pass(被跳过规则过滤) |
| 反例 3(表格) | `\| 项目管理 \| 质量审核 \| 工作程序 \| 风险控制 \|` | pass(被表格跳过规则过滤) |
| 反例 4(段落含连接词) | "通过质量管理体系,我们对项目实施全过程的审核,确保工作程序符合标准方法,从而控制风险并提高质量"(7 关键词 50 字,有连接词,跨度 > window) | pass(window=30 字内只命中 < threshold) |

每条 case 用 inline 字符串 + 含真实评分项关键词的 mini matrix_rows fixture,断言准确。

**问 3:误报时用户怎么处置?**

三档 escape(从轻到重):

1. **行级忽略**:在 markdown 里被误报段落上方加 `<!-- nolint:stitching -->` 注释。check_content_keyword_stitching 检测这种行,跳过其后第一个非空段落。适用单点误报。
2. **CLI 调阈值**:`./run_script.bat check_chapter.py ... --stitching-threshold-warn 5 --stitching-threshold-fail 6` 全局放宽。适用整篇误报多。
3. **CLI 调窗口**:`--stitching-window 40` 把 30 字窗口拉宽到 40 字(阈值不变,要求关键词更密集才命中)。适用文本密度普遍高的项目类型(如 工程 项目)。

CLI flag 默认值印在 main 的 `--help`,用户被误报后看 help 知道怎么改。docs/FAQ.md 加一条 Q&A "缝合句检测误报怎么办"(本期同 commit 加,scope 内)。

### 2.3 跳过规则(避免枚举式堆叠误报)

沿用 check_whitespace_rule(L320-)已有的跳过逻辑:

- 跳过代码块(``` 围起来的)
- 跳过表格行(以 `|` 开头)
- 跳过 ATX heading(以 `#` 开头)— 标题级由 check_title_keywords 已有 cheat 检测覆盖
- 跳过列表行(`- ` / `1. ` 引导)— 枚举式堆叠是合理用法
- 跳过图占位符 `**【图 X.X:...】**`
- 跳过 HTML 注释 `<!-- ... -->` — 含 nolint 标记的注释行被识别后应用其规则
- **新增**:跳过被 `<!-- nolint:stitching -->` 标记的下一个非空段落

正文级缝合检测**只在自由段落**生效,不在结构化区域。

### 2.4 报告输出格式

借鉴 check_title_keywords L161-166 的格式:

```
[8] 正文级缝合句  失败
        L42 段落: 命中 5 关键词(质量/审核/管理/控制/程序)跨度 24 字
        见 ai_output_rules.md R7 缝合句作弊扩展检查
        如属正常专业段落,加 <!-- nolint:stitching --> 单点忽略,或调
        --stitching-threshold-fail / --stitching-window
```

## 6. 风险

| 风险 | 缓解 |
|---|---|
| 阈值 4/5 在 demo 上误报 → 不能给 baseline 用 | Phase 2 commit 1 内 demo 烟测,误报多就调到 5/6 或 4/6 等;阈值最终值在 Phase 3 报告说清楚 |

| 跳过规则太严 → 真缝合句也被跳过 | 用正例 1(纯文本段落)测必须命中;不能因为 skip 规则把所有命中都干掉 |

| 实现"前后段相邻" nolint 时遇空行/HTML 注释行边界 | "nolint 后第一个非空、非注释段落"——边界用 test 验 |
