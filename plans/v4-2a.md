# plans/v4-2a.md — V4-2a B 智能匹配·链路打通

## 任务定义

V4-2(B 智能匹配)已拆为 V4-2a(链路打通)+ V4-2b(元数据精化 + rationale)。本任务 = V4-2a:让 AI 在 B 模式 extract 阶段写 intermediate.json 时 awareness 库内候选清单 + 本 Part 关联评分项,使 AI 从真实候选中选材(消除当前"库空仍盲写 asset_query"的行为),而非凭招标文件原文盲写。

当前链路断在 2 点(Phase 0 实测):
- scoring_matrix → B 模式 prompt:B 模式 extract/fill/run 全程不读评分矩阵(grep 0 命中)
- assets 库 → B 模式 prompt:provider 只在 fill 阶段被调,不参与 extract 阶段 AI 撰写;AI 写 asset_query 时不知库里实际有什么(实测:assets 目录空时 AI 仍盲写 asset_query{name:营业执照})

## 已定裁定

- awareness 机制 = 甲:脚本扫 assets 文件系统产候选清单 → 注入 AI 撰写源 → AI 据评分项 + 候选选材。脚本只做"收集候选 + 透传"(无语义判断),AI 做"选哪个"(语义)。职责边界沿 R10 + 决策 #5。
- 边界硬约束:V4-2a 只扫文件系统可见信息(类别=目录名 / company_id=目录名 / 文件名 / year=文件名 YYYYMMDD_ 前缀),不读 frontmatter 元数据(14 字段留 V4-2b)。设计意图:用户按目录结构丢文件进 assets/ 即可用,零元数据负担。
- 占位 = 甲:AI 在候选清单里选不到匹配(库空 / 无匹配评分项的候选)时,asset_query 必须显式标"库内无匹配候选 / 待用户补料"状态,不盲写不存在的 name。复用 V4-4 占位 R10 范本(没有就显式喊出来,不假装有)。
- rationale 剔到 V4-2b:V4-2a 只记"选中了哪条"(inventory_match,链路打通必需),不写"为什么选"(rationale 撰写 + selection_rationale 产物 = V4-2b)。
