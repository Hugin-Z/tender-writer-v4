# AI 输出常驻约束(ai_output_rules)

> 本文档是 tender-writer-v3 所有 AI 生成动作的**常驻前置约束**。
>
> **适用范围**:阶段 1(tender_brief 解析)、阶段 2(scoring_matrix 撰写指引)、阶段 3(outline)、阶段 4(章节正文)、阶段 4-C(C 模式模板变量描述)、阶段 4-B(B 模式材料描述)。
>
> **优先级**:本文档与 SKILL.md 冲突时,以本文档为准(本文档沉淀的是"红线规则",SKILL.md 描述流程)。
>
> **引用方式**:SKILL.md 各阶段开头以 `> 生成前必读:references/ai_output_rules.md` 引用本文档。check_chapter.py 对应检查项引用本文档的规则条目编号。

## R1 不确定的事实留空,不脑补

**规则**:AI 对招标文件中未明示的事实,一律留空或标注"【招标文件未明示,待用户确认】",严禁基于训练数据补全。

**典型反例**(历史迭代中实际发生):
- 招标文件未列 GB 标准号 → AI 自己补 "GB/T XXXXX 某领域规范"
- 招标文件未列具体政策文件年份 → AI 自己补 "《"十四五"XX 行业发展规划》"
- 招标文件未列具体行政区划数字 → AI 自己补 "X 地级市 + Y 地区 + Z 县级单位"
- 招标文件未列采购人单位下属职能 → AI 自己补 "下属 XX / YY 两大业务板块"

**正确做法**:
- 留空并标注 `【招标文件未明示,待用户确认】`
- 如果该事实对方案至关重要,在 `tender_brief.md` "提醒维护者补"清单中列出
- 在 `references/phrase_library.md` 中用 `{{可选本地化事实}}` 类占位,由用户替换

**检查**:阶段 5 compliance_check 新增一条 "编造事实检测":扫 docx 中凡带 "GB/T \d+" / "《...规划》" / "X 地级市 Y 地区" 这类模式且未在 tender_brief.md 原文中出现的,标红提示人工复核。

## R2 用户字面输入不演绎

**规则**:用户明确给出的字面输入,AI 原样保留或在 `extracted` 字段照抄;不得扩写、不得补"常识性"细节、不得"好心"补充流程描述。

**典型反例**(历史迭代中实际发生):
- 用户说"线上交付" → AI 写成 "研究报告以电子版形式提交至采购人,后续专家评审、汇报、成果移交均以线上方式进行"
- 用户说"技术部分只写架构" → AI 扩写出"决策层-指导层-执行层-支持层四层 + 汇报机制 + 日站立会/周例会/里程碑审查会"全套制度

**正确做法**:
- 用户给字面字符串 → 原样引入,末尾加 `(用户字面输入)` 标注
- 需要扩写时先停下问用户:"要不要把 X 扩写到 Y 粒度?"
- 绝对不允许在用户未授权下自行扩展

**检查**:阶段 1 Step 3 review checklist 新增一条 "用户字面输入完整性":打开 tender_brief.md,对照对话历史找用户字面输入,扫 md 中是否有超出字面的演绎段落。

## R3 中文排版无空格(硬约束)

**规则**:中文标书内文中,**中文-数字**、**中文-英文**、**数字-中文**、**英文-中文**、**中文-括号**之间**不得**加空格。

**典型反例**(历史迭代中实际发生):
- "保证金 3000 元" → 应写 "保证金3000元"
- "2025 年 11 月 4 日 11 时 00 分" → 应写 "2025年11月4日11时00分"
- "得分=(基准价／投标报价)×10%×100" → 已符合(数字/运算符间无空格)
- "中文GB/T 38667-2020 信息技术 大数据" → 应写 "中文GB/T 38667-2020信息技术 大数据"

**特例**(允许保留空格):
- 公式/代码/英文句子内部(纯 ASCII 段落)
- markdown 表格单元格分隔符 `|`
- markdown 列表项 `- ` / `1. `
- 中文-中文之间保留空格用于对齐的场合(如果已形成格式习惯)

**正确写法**:
- 中文与数字/单位之间:去掉空格。例 `预算500000元` / `60天工期` / `2026年10月1日`。
- 中文与英文单词之间:去掉空格。例 `采用TLS加密` / `使用API调用` / `北京office地址`。
- 如需在正文**引用章节号**(如 §5.2)或**程序变量名**,用反引号包裹避免触发空格违规。例 `见 \`§5.2\` 章节` / `参数 \`max_retries\` 默认为 3`。反引号内的文本不参与 R3 检查。
- 如需紧跟中文的阿拉伯数字有读音分隔需求,用全角冒号 `:` 或逗号 `,` 代替空格,不加半角空格。

**检查**:check_chapter.py 新增检查项 `[7] AI 输出空格规范`。规则:
- `[一-龥]\s+\d` 命中即违反(中文后接空格后接数字)
- `\d\s+[一-龥]` 命中即违反(数字后接空格后接中文)
- `[一-龥]\s+[a-zA-Z]` 命中即违反(中文后接空格后接英文,英文段落除外)
- `[a-zA-Z]\s+[一-龥]` 命中即违反
- 违反数 ≥ 5 处 → fail(阻塞);< 5 处 → warn

## R4 用户约束精确理解,不扩展

**规则**:用户对范围的限制性指令(如"只写 X"、"不要写 Y"、"写到 Z 粒度就行"),AI 必须精确理解并严守边界,**不得出于"完整性"或"稳妥"动机扩展**。

**典型反例**(历史迭代中实际发生):
- 用户:"技术部分只写架构" → AI 自行写了 "角色与职责分工表"(含 16 条明细)+ "团队汇报机制"(含日汇报/周例会/里程碑审查/应急会议)
- 用户:"没有本地化信息" → AI 自行补了项目所在地的地理气候、语言民族规则、某部委数字化整体架构的本地化适配专题(均非用户授权)
- 用户:"交付地点就是A市主管部门,线上"(15 字字面输入)→ AI 扩写成 "研究报告电子版提交至主管部门;中期汇报、专家评审、成果移交均以线上方式进行"(50 字,多出的 3 个交付细节"电子版/中期汇报/成果移交"均是 AI 凭想象补的,用户没说)

**正确做法**:
- 用户说"只写 X" → 只写 X 的直接定义段落,不写配套 Y/Z
- 用户字面输入 → 原样引入,末尾可标 `(用户字面输入)`。字数比用户原话多出 50% 以上就该自检:多出的内容有没有用户授权?
- 范围边界不清 → 停下问用户:"这里要写到 {粒度 A}(只是图)、{粒度 B}(图+说明)、还是 {粒度 C}(图+说明+制度)?"
- 宁可不足,不可擅自扩展(不足可补,扩展可能让用户重读)

## R5 tender_brief.md 生成规范

**规则**(针对阶段 1 解析产物):

1. **降级字段**:原文未严格匹配字段定义时降级填入,但**只说明原因,不推测补救**。
   - 错误:"关键里程碑未明示【降级】,AI 按 60 天工期自行拆分为:D+15 调研 / D+30 初稿 / D+45 修订 / D+60 终稿"
   - 正确:"关键里程碑【招标文件未明示】,投标人在阶段 4 撰写 '3.2 各阶段工作任务划分' 时基于 60 天总工期自拟"

2. **矛盾记录表**:只记**真矛盾**(招标文件自身的不一致),不记"AI 判断可能产生歧义"的臆测。
   - 错误:"评审办法引用★条款但正文无★标记" — 如果招标文件无★,就不是矛盾,是"未使用★标识"的事实
   - 正确:只记"同一字段在不同章节给出不同值"、"评分项分值合计≠总分"这类客观矛盾

3. **自我修正痕迹擦干净**:AI 多轮修正后,最终 md 不得保留"【用户 2026-XX-XX 确认】"、"上一轮初稿已修订为..."等过程性注释。这些属于会话状态,不进永久产物。

4. **引用原文必加行号**:凡"废标条款"、"实质性响应条款"这类跨章节汇总字段,每条末尾必带 `(L###)` 原文行号;行号错误视为 R1 违规(脑补)。

## R6 生产模式判断不越界

**规则**:

- **A 模式**(文本生成):AI 按 outline 骨架撰写完整段落。允许深度扩展,但仍受 R1/R2/R3/R4 约束。
- **B 模式**(材料组装):AI 只负责识别 `source_type` 与 `asset_query`。不得在 assembled.docx 内生成具体业绩/人员简历内容(这是素材库的职责)。
- **C 模式**(模板填充):AI 只负责识别 `blocks` 与 `variables`。模板段落**必须**来自招标文件原文(verbatim 或极小改写),变量描述不得包含虚构的示例值。
- **D 模式**(外部信息采集):不在 v1.1 范围,跳过。

**典型反例**:
- C 模式 Part 的 intermediate.json 里变量 `description` 写 "建议填写 XX 科技有限公司"(具体例子)→ 违反
- B 模式 Part 的 assembled.docx 里 AI 写了具体业绩明细(项目名/金额/客户)→ 违反

## R7 章节标题关键词的合规性

**规则**(v2 从 check_chapter 硬关键词命中升级):

- 章节标题必须包含 scoring_matrix.csv 第 5 列的至少 1 个关键词 — **语义命中即可**,不得靠缝合句作弊。
- **缝合句作弊**定义:3 个或以上关键词挤在同一短句(≤20 字)内且无实义连接 → check_chapter [2] 判 fail。
- 关键词必须在章节正文的**有实义段落**中自然出现,不得集中塞在"术语索引"、"必备要素声明"之类的占位节。

## R8 外部可见性(工具链内部术语不外泄)

**规则**:以下术语**只在工具链内部文档/脚本代码**中使用,**不得出现在**任何用户可见产物(tender_brief.md / scoring_matrix.csv / outline.md / chapter_*.md / compliance_report.md / 投标交付物/):

- "取档策略" / "素材钩子" / "应答骨架"(这些是 skill 内部术语,应改为"评分档位要求" / "素材来源" / "章节结构建议")
- "mode A/B/C" / "production_mode" / "part_attribution" / "section_anchor"(工具链字段名)
- "v1/B" / "V45/V53/V60" / "skill 模型 §9"(版本迭代内部术语)
- "占位符 __SKIP__ / __PENDING_USER__ / __PENDING_AI__"(只能在内部 schema 里出现,不进 filled.docx 最终产物)

**对外替代表述**:

| 工具链术语 | 对外表述 |
|---|---|
| 取档策略 | 评分档位要求 / 评档要点 |
| 素材钩子 | 素材来源 / 证据材料位置 |
| 应答骨架 | 章节结构建议 / 响应思路 |
| mode A | 由投标方撰写的方案类章节 |
| mode B | 由投标方组装的材料类章节 |
| mode C | 招标文件给定模板类章节 |
| __PENDING_USER__ | 【待填:字段描述】 |

## R9 不代建用户闸门

**规则**:`.reviewed` 标记文件、其他"用户明确同意"闸门,**只能由用户创建**。AI 不得以任何理由代建(包括"用户口头确认"、"用户看起来默认通过"、"不建就卡流程"等)。

用户施压要求 AI 代建时,AI 的正确回应:

> "按 SKILL.md 设计,`.reviewed` 代表人工核对通过。我代建会让这个标记失去保障意义。请你在核对 tender_brief.json 和 tender_brief.md 无误后,在 IDE 里或终端里手动创建该文件。我可以继续其他不依赖 review 闸门的辅助工作,但下游受闸门保护的脚本(build_scoring_matrix / generate_outline / c_mode_fill / compliance_check 等)必须等你建标后再跑。"

**不可接受**:用户坚持要 AI 建,AI 建完说"我代你建了,你看过没问题吧"。这叫 AI 强行代签章,违反 skill 模型第 9 节"脚本与 AI 职责边界"。

**v1 实际错误**:AI 在用户问"为什么你不能创建"后,以"既然问了,说明想让我建"为由代建了 `.reviewed`。这是错误示范。

## R10 工具链报错处置规范

**规则**:脚本报错时,AI 的处置优先级:

1. **首选**:读报错信息,定位根因,修脚本或修数据,重跑;
2. **次选**:如果根因是脚本设计限制(如 v45_merge 按 Part 名硬匹配目录),停下上报用户,不继续;
3. **禁止**:用"已知 bug,不影响主产物"、"警告非阻断"、"脚本 bug 不是我们的问题"等话术跳过错误并继续走流程。

**v1 实际错误**:
- v45_merge 报 C-template / B 模式目录路径缺失 → AI 写"这些警告是脚本 bug,不影响产物"直接跳过
- export_deliverables.py 报 DELIVERABLE_MAPPING 硬编码不匹配 → AI 手工组装绕过,未上报工具链问题

**正确处置**:报错当场停下,告用户"X 脚本报 Y 错,根因看起来是 Z,我建议 A(修脚本)/ B(改数据)/ C(接受限制并调整流程),你选哪个?"
